# Multi-agent orchestration: 5 new agents, database layer, knowledge base, protocols, and plugin system

Phase 3 adds the full multi-agent expansion to J.A.R.V.I.S: a database layer (SQLAlchemy with MySQL/SQLite fallback), five new specialized agents (web search, data analyst, scheduler, writer, life assistant), a TF-IDF knowledge base, an inter-agent communication protocol, a priority task queue, and a plugin loader. The orchestrator gains regex-scored routing for the new task types, checked before the existing code/research paths. The approach heavily uses graceful degradation — every external import is wrapped in try/except, every database operation returns a success/error dict, and the system remains functional even when MySQL, sklearn, or duckduckgo-search are absent.

**Watch for:** `create_all_tables()` signature mismatch when called from `KnowledgeBase.__init__` (confirmed bug — will TypeError at runtime), `health_check()` leaking database credentials in the URL field (confirmed), `data_tools` accepting arbitrary file paths with no sandboxing (confirmed), and routing ambiguity where a single common-word match on "data" or "tulis" can steal a task from the code/research agents (likely).

## High-level view

The database layer provides connection pooling via SQLAlchemy with an automatic SQLite fallback, six ORM models, and an idempotent migration function. The health check endpoint exposes the full connection URL including credentials.

Five tool modules follow a uniform pattern: module-level singleton `_get_db_manager()`, graceful import guards, and dict-return contracts. The data tools accept user-provided file paths without constraining them to a sandbox directory, which means an LLM-driven agent could read any file the process can access.

The knowledge base combines SQL keyword search with TF-IDF cosine similarity, merging and deduplicating results. Its `__init__` calls `create_all_tables(self._db.engine)` but the migration function accepts zero arguments — this will raise a TypeError on first use.

The orchestrator routes new task types before code/research using a score-based approach: if any new-pattern score reaches 2+, it wins unconditionally; if it's 1 and ties with code/research, the new agent wins by default. Single-keyword matches on common words can route unexpectedly.

The inter-agent protocol and priority task queue are standalone infrastructure not yet wired into the orchestrator — building blocks for future delegation workflows.

<summary>Details</summary>

## `create_all_tables` arity mismatch in KnowledgeBase

`knowledge_base.py` line 62 calls `create_all_tables(self._db.engine)`, passing the engine as a positional argument. The function in `migrations.py` is defined as `def create_all_tables() -> Dict[str, Any]` — it takes no parameters and internally creates its own `DatabaseManager()` to get an engine. This call will raise `TypeError: create_all_tables() takes 0 positional arguments but 1 was given` the first time `KnowledgeBase()` is instantiated with a live database.

The simplest fix is to call `create_all_tables()` with no arguments (it already handles engine acquisition), or to add an optional `engine` parameter to `create_all_tables` and skip creating a new DatabaseManager when one is provided.

## Credential exposure in `health_check`

`DatabaseManager.health_check()` returns `"url": str(self.engine.url)`. For MySQL connections, this URL is of the form `mysql+pymysql://root:password@localhost:3306/jarvis_db`. Any caller that logs, serializes, or surfaces this dict to an API exposes credentials. SQLAlchemy provides `engine.url.render_as_string(hide_password=True)` which replaces the password with `***`.

## Unsandboxed file access in data_tools

`_validate_file_path` calls `os.path.abspath(file_path)` and then only checks existence — no directory whitelist, no symlink resolution, no restriction. Since these tools are invoked by an LLM agent processing user prompts, a prompt injection attack or a misclassified request could instruct the agent to call `read_csv_file("/etc/passwd")`. The data_tools extend the surface area to structured data reading with pandas, which can also follow URLs if `pd.read_csv` receives a URL-like string.

## Task routing score tie-break

The orchestrator computes scores for all five new patterns plus code and research. The decision logic:

```
if best_new_score >= 2: return best_new_type
...compute code/research scores...
if best_new_score >= 1 and best_new_score >= code_score and best_new_score >= research_score:
    return best_new_type
```

The `>=` on the final condition means a 1-1 tie between (say) `write_score` and `code_score` routes to the writer agent. The edge case is prompts where a single new-pattern word matches and code/research also scores exactly 1 — the new agent wins by default. For inputs like "tulis kode python sorting", the code patterns should win (scoring 2+ from "kode" + "python"), but borderline cases like "search API documentation" would route to web_search instead of research.

## Protocol and queue: unintegrated infrastructure

`AgentProtocol` and `TaskQueue` are fully implemented but not referenced from the orchestrator or any agent. They provide delegation primitives (`send_message`, `delegate_task`, `aggregate_results`) and a heapq-backed priority queue, but the actual orchestrator still routes directly to agent `.run()` calls. Noting this so the next phase can wire them in rather than building parallel infrastructure.
