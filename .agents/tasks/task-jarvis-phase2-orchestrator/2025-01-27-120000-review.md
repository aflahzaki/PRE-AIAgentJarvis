# Multi-agent orchestrator with dual-provider routing

This change introduces a complete Phase 2 for the J.A.R.V.I.S system: a multi-agent architecture with local (Ollama) and cloud (Groq) LLM providers, heuristic-based task routing by difficulty, a self-healing agent loop with text-based tool calling, conversation memory with naive summarization, file/execution tools, and a Rich-powered interactive REPL. The approach uses raw OpenAI SDK to talk to both Ollama and Groq through their OpenAI-compatible endpoints, avoiding the Phidata dependency of Phase 1. Old files are untouched; all new code lives under `core/`.

**Watch for:** The self-healing loop confounds tool-calling retries with provider-error retries, making it possible for a single provider timeout to consume all 5 retry slots without ever attempting tool execution (confirmed). The python executor has no sandboxing—the LLM-directed agent can write and execute arbitrary files as the host user (confirmed). The task classifier and difficulty classifier are fully independent systems with duplicated regex lists that can disagree, routing e.g. a "security" query to the researcher agent but the small local model (confirmed).

## High-level view

The provider layer is thin and clean: an abstract `BaseProvider` with `chat_completion` and `is_available`, backed by Ollama (local, unlimited, offline) and Groq (cloud, rate-limited). Both wrap the OpenAI SDK. The Groq provider implements client-side rate limiting by timestamp tracking, but the health check (`is_available`) never actually verifies the API key is valid—it only checks that the key string is non-empty and we haven't self-imposed a cooldown.

The routing layer has a split-brain problem. `ModelRouter.classify_difficulty` decides EASY/MEDIUM/HARD for provider/model selection, while `Orchestrator.classify_task` independently decides code/research/simple for agent selection. These two classifiers operate on partially-overlapping regex patterns and make independent decisions. A "design a microservices architecture" input is HARD (routed to Groq 70B) *and* research (researcher agent)—that's coherent. But "security audit" is HARD (Groq 70B) yet hits `research` in the orchestrator, which hands it to a ResearcherAgent that has no tools—appropriate here, but the coupling is accidental rather than designed.

The agent layer's key design choice is text-based tool calling via `[TOOL_CALL]...[/TOOL_CALL]` markers instead of native OpenAI function calling. This ensures compatibility with all providers regardless of whether they support the `tools` parameter. The trade-off is fragile parsing—the LLM can produce malformed JSON or forget the tags entirely, and there's no validation beyond `json.loads`.

The self-healing loop in `BaseAgent.run()` counts both provider errors *and* successful-but-tool-calling iterations against the same retry budget. A transient network failure burns a retry slot, leaving fewer iterations for the actual write-execute-fix cycle that self-healing is designed to serve.

The python executor runs arbitrary code without any sandboxing, path restriction, or resource limit beyond a 30-second timeout. Since the coder agent writes files via `write_file` and then executes them via `execute_python`, an adversarial or confused LLM response could write and run destructive code on the host filesystem.

Memory management uses a fixed-size buffer with a crude summarization strategy: when the buffer fills, the oldest half is compressed into truncated `[role]: first-200-chars` fragments. This loses tool-call context and code snippets that don't fit in 200 characters—problematic for multi-turn coding sessions where the agent needs to recall what it previously wrote.

<details>
<summary>Issues (7)</summary>

1. **Retry budget conflation** — Provider errors and tool-calling iterations share the same counter. Separate them so transient failures don't eat into the self-healing budget.
2. **Unsandboxed code execution** — The coder agent can write and execute arbitrary Python as the host user. Add path whitelisting and consider subprocess sandboxing (e.g., `--nosite`, restricted env vars, or a container).
3. **Dual-classifier disagreement** — `ModelRouter.classify_difficulty` and `Orchestrator.classify_task` use independent regex sets that can produce incoherent routing. Unify them or make one delegate to the other.
4. **Groq API key validation gap** — `is_available()` returns True if the key string is non-empty, even for `"your-groq-api-key-here"` when the placeholder accidentally leaks into `.env`. The placeholder check exists in `__init__` but any non-placeholder garbage key passes the health check.
5. **Memory truncation loses tool context** — Summarization truncates messages at 200 characters, discarding code snippets and tool results from earlier turns. Coding tasks that span many turns will lose critical context.
6. **Tool-call parsing fragility** — No recovery path when the LLM produces a partial `[TOOL_CALL]` tag, an unclosed tag, or valid JSON that's missing the `name` field. The loop silently drops these and may terminate without a final answer.
7. **File tools have no path restriction** — `write_file` and `read_file` accept absolute paths with no whitelist, so the LLM could overwrite system files if the host user has permission.

## Self-healing loop conflates error types

In `base_agent.py` the `run()` method uses a single `retries` counter for both provider failures and tool-calling iterations:

```python
while retries < self.max_retries:
    response = self._call_provider_with_tools(messages)
    if not response.success:
        retries += 1
        continue
    ...
    # tool calls executed
    retries += 1
```

A coding task that hits one network timeout and then needs 4 tool-call rounds (write, execute, fix, re-execute) will exhaust the budget. The fundamental issue is that tool iteration is *expected progress*, not failure—it shouldn't consume the same budget as error recovery.

## Unsandboxed Python execution

`python_executor.py` runs scripts with the same interpreter (`sys.executable`) and no restrictions beyond a 30-second timeout. The `cwd` defaults to the script's directory, and there's no restriction on what files the script can touch. Combined with `write_file` having no path whitelist, an agent that hallucinates a command like `os.system("rm -rf /")` in generated code would execute it.

The `.py` extension check on `execute_python` is trivially bypassed via `execute_python_code`, which writes to a temp file (always ends in `.py`). There's no `--isolated` or restricted environment.

## Dual classifier disagreement

The orchestrator has two independent classification steps:

```
Orchestrator.classify_task()  ->  code / research / simple  (picks agent)
ModelRouter.classify_difficulty()  ->  EASY / MEDIUM / HARD  (picks provider+model)
```

Both use regex pattern matching but with different keyword lists. `HARD_PATTERNS` in the router includes "security audit" and "machine learning"—so does `RESEARCH_PATTERNS` in the orchestrator. But the orchestrator's `CODE_PATTERNS` includes "python" and "execute", which overlap with `MEDIUM_PATTERNS` in the router. The two systems agree *usually*, but there's no guarantee.

Concrete disagreement: input "explain python decorators in detail" matches CODE_PATTERNS (contains "python") in the orchestrator -> code task -> CoderAgent. But it also matches HARD_PATTERNS ("explain in detail") in the router -> Groq 70B. The user gets a coder agent for what's really a conceptual question, wasting tool-calling overhead.

## Groq availability check doesn't validate the key

```python
def is_available(self) -> bool:
    if not self.client:
        return False
    if time.time() < self._rate_limited_until:
        return False
    return True
```

The `__init__` excludes the literal placeholder `"your-groq-api-key-here"`, but any other invalid key (typo, expired, wrong account) will pass `is_available()` and only fail at request time. The system will route to Groq, fail, and return a generic error—rather than falling back to Ollama proactively.

## Memory summarization truncation

`_summarize_old_messages` compresses messages to `content[:200]`. In a coding session, assistant messages routinely contain full file contents or multi-line code. Truncating to 200 characters loses the actual code that was written. When the agent later needs to recall what it previously implemented (e.g., to fix a bug in its own output), the summary provides only a meaningless prefix.

The summarization also keeps only the last 20 entries of the summary itself (`summary_parts[-20:]`), so long sessions lose context progressively.

## File tools accept arbitrary paths

`read_file` and `write_file` operate on `os.path.abspath(file_path)` with no restriction. An LLM producing `write_file(file_path="/etc/crontab", content="...")` would succeed if the process user has permissions. For a development tool intended to run on the user's machine, this is a meaningful attack surface—especially since the tool-calling is driven by LLM output that could be influenced by prompt injection in read files.
