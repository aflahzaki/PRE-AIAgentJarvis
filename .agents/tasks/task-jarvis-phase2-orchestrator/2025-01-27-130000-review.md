# Multi-agent orchestrator with dual-provider routing (v2)

All seven issues from the first review have been addressed. The retry loop now uses separate budgets for provider errors (max 3) and tool-calling iterations (max `max_retries`). The python executor validates workspace-bounded paths, uses `-I` (isolated mode), and strips sensitive env vars. The orchestrator's `classify_task` now consults the router's difficulty classification to resolve disagreements. Groq's `is_available()` rejects short keys and common placeholder patterns. Memory summarization allows 500 chars for tool/code content and keeps 30 summary entries. Tool-call parsing warns on mismatched tags, missing `name` fields, and malformed JSON. File tools enforce a `JARVIS_WORKSPACE`-bounded path restriction.

**Watch for:** The workspace restriction relies on `os.getcwd()` at import time if `JARVIS_WORKSPACE` is unset — a user running the REPL from a parent directory like `$HOME` would grant the LLM access to the entire home tree (confirmed). The non-greedy regex in tool-call parsing will fail on nested JSON containing `}` (e.g., arguments with dicts-in-dicts), silently dropping valid tool calls (confirmed). The temp-file directory for `execute_python_code` creates `.tmp/` within the workspace but never cleans stale files across sessions (likely).

**Verdict**: APPROVED

## High-level view

The workspace-root security boundary gates every file tool through a single `_validate_path` helper, and the executor mirrors the logic. The weakness is the default value when `JARVIS_WORKSPACE` is unset — `os.getcwd()` is evaluated at module import time, which on a typical user's Windows machine will be wherever they launched the script from. If that's `C:\Users\username`, the LLM gets read/write over the entire profile.

Provider errors now decrement a dedicated counter and `continue` without touching `tool_iterations`, fully resolving the original budget-conflation issue.

The dual-classifier coherence fix has `classify_task` call `self.router.classify_difficulty` and use the result as a tiebreaker: if difficulty is HARD and any research signal exists, research wins. This eliminates the pathological "explain python decorators" case where a language keyword routed conceptual questions to the coder agent.

The tool-call parser's `re.DOTALL` non-greedy pattern `\{.*?\}` cannot match arguments containing nested objects because `.*?` stops at the first `}`. An LLM producing `{"name": "write_file", "arguments": {"path": "x", "options": {"a": 1}}}` will get a parse failure. The warning is logged but the call is silently lost.

<summary>Issues (3)</summary>

1. **Workspace default is too broad** — If `JARVIS_WORKSPACE` is unset, `os.getcwd()` at import time becomes the boundary. On Windows, launching from `C:\Users\name` grants full home-directory access to the LLM. Default to the script's own directory or a dedicated `workspace/` subfolder instead.
2. **Nested JSON breaks tool-call regex** — The non-greedy `\{.*?\}` pattern fails on arguments containing nested dicts. Use a bracket-counting parser or `json.JSONDecoder.raw_decode` to extract the JSON object reliably.
3. **Temp directory accumulates files** — `execute_python_code` creates `.tmp/` but only cleans the file from the current invocation. Across sessions, stale `.py` files accumulate. Add startup cleanup or use `tempfile.TemporaryDirectory` scoped to the session.

## Workspace default when JARVIS_WORKSPACE is unset

Both `file_tools.py` and `python_executor.py` define their workspace root identically:

```python
_WORKSPACE_ROOT = os.path.abspath(
    os.getenv("JARVIS_WORKSPACE", os.getcwd())
)
```

This is evaluated once at import time. The `.env.example` does not include a `JARVIS_WORKSPACE` variable, so typical users will rely on the default. On Windows, double-clicking the script or launching from a terminal in the user's home directory means `os.getcwd()` is `C:\Users\<name>` — and the entire home tree becomes the allowed workspace.

A safer default would be `os.path.dirname(os.path.abspath(__file__))` (the `core/tools/` directory) or a dedicated `workspace/` subfolder relative to the project root. Alternatively, `run_jarvis_v2.py` could set `JARVIS_WORKSPACE` to the J.A.R.V.I.S_system directory before any imports.

## Non-greedy regex drops nested-object arguments

`_parse_tool_calls_from_text` uses:

```python
pattern = r'\[TOOL_CALL\]\s*(\{.*?\})\s*\[/TOOL_CALL\]'
```

The `.*?` (non-greedy with `re.DOTALL`) matches the *shortest* string ending in `}`. For a tool call like:

```
[TOOL_CALL] {"name": "write_file", "arguments": {"path": "x.py", "content": "a = {1: 2}"}} [/TOOL_CALL]
```

The regex captures up to the first `}` that satisfies the non-greedy match, `json.loads` fails on the partial string, and the warning fires. The tool call is lost with no recovery.

A bracket-counting approach or using `json.JSONDecoder().raw_decode(content, idx)` after finding `[TOOL_CALL]` would handle arbitrary nesting. Alternatively, match everything between `[TOOL_CALL]` and `[/TOOL_CALL]` greedily (since the closing tag is unambiguous) and then parse the captured text as JSON.

## Temp file cleanup gap

`execute_python_code` creates files in `<workspace>/.tmp/`:

```python
workspace_tmp = os.path.join(_WORKSPACE_ROOT, ".tmp")
os.makedirs(workspace_tmp, exist_ok=True)
temp_file = tempfile.NamedTemporaryFile(..., dir=workspace_tmp)
```

The `finally` block removes the file from the current invocation. But if the process crashes mid-execution (OOM kill, Ctrl+C during `subprocess.run` before `finally` runs), the temp file persists. Over many sessions, `.tmp/` accumulates stale scripts. This is a minor hygiene issue rather than a security concern, since the files are within the workspace and contain only code the LLM already generated. A startup sweep of `*.py` files older than 1 hour in `.tmp/` would suffice.
